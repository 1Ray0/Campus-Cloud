## 三、逐專案詳細檢查

### 1) ai-template-recommendation

路由處理：
- 子專案有 chat/recommend 雙端點（含 /api/v1 別名）。
- 主專案已有對應 /ai/template-recommendation/chat 與 /recommend。
- 子專案的 /resource-options、/lxc/templates、/vm/templates、/catalog/preview、/ui-config 這類輔助端點，主專案未以同路由型態直接暴露。

提示詞模板：
- 子專案 prompt 與主專案 prompt 內容一致度高（可視為已移植）。

前端：
- 主專案前端已在 ApplicationRequestPage + AiChatPanel 串接模板推薦 API。
- 子專案 static/index.html 測試頁並未成為主專案正式 UI 路由。

依賴：
- 子專案版本來有獨立設定與測試入口。
- 主專案版改為內嵌透過 proxmox service 與內部 node/resource option 流程取得資料，不再需要啟獨立 template 服務。

判定：
- 核心能力已同步。
- 若你要「完整替代子專案 API 面」，仍有少量輔助端點差異。

## 四、你現在最需要注意的依賴狀況

1. 仍需獨立子服務才能用完整功能
- ai-pve-log：是，基本上全部功能都在獨立服務。
- ai-pve-placement-advisor：若你要 explain/metrics/preview 等完整能力，仍是。

2. 主專案內可直接使用
- ai-template-recommendation 核心推薦流程：可。
- ai-teacher-judge 核心流程：可。

3. 套件/環境風險
- ai-teacher-judge 相關解析/匯出依賴在主專案 pyproject 尚未明列，建議列入正式依賴治理。

## 五、建議優先順序（依你職責）

P0（先做）
- 補齊 ai-teacher-judge 在主專案的依賴宣告一致性（pdfplumber、python-docx、openpyxl 等）。

P1（第二階段）
- 決策 ai-pve-placement-advisor 的產品邊界：
  - 若要主專案完全承接，補 /analyze、/metrics、/explain、/sources/preview 對外路由與前端頁。
  - 若保留獨立服務，需正式標註其為外掛，避免誤判為已整合。

P2（第三階段）
- 決定 ai-pve-log 是否納入主專案：
  - 納入：需規劃 API namespace、權限、前端入口、compose 服務與監控。
  - 不納入：應文件化為獨立運維域，避免跨團隊認知落差。
